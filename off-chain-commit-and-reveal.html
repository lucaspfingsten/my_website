Off-Chain Commit & Reveal

Off-chain commit & reveal protocols order transaction in two phases. In the first phase, users send their transactions to a committee. The committee then agrees on an order through consensus. In the second phase, once the order is agreed upon, the committee decrypts the transaction with their threshold signatures.

Positives

  • Good Scope
  • Minimal Jostling
  • No increased delay
  • No decreased goodput

Negatives

  • Increased cost
  • Reduced decentralization
  • Security in case of byzantine committee

Methods

HoneyBadgerBFT

The HoneyBadgerBFT protocol operates by allowing nodes in a distributed system to reach consensus even under asynchronous conditions, where there are no guarantees on message delivery times. It employs a novel atomic broadcast algorithm to ensure all honest nodes agree on the same set of transactions, without relying on timing assumptions. This is achieved through a combination of threshold cryptography for confidentiality and reliability, and randomized protocols to cope with Byzantine faults. Transactions are encrypted using the public key and ordered by the committee, with a two-thirds majority assumed to be honest. The committee then collectively decrypts the messages using their threshold signatures, ensuring secure and fair transaction processing.


For more information read the paper:
The Honey Badger of BFT Protocols

Helix

The Helix protocol addresses transaction reordering in distributed networks through random selection and ordering of transactions using correlated sampling. It also incorporates threshold encryption to obscure transaction sources, enhancing fairness and reducing manipulation risks in transaction processing.

How to Share a Secret

The "How to Share a Secret" protocol incorporates polynomial interpolation for secure processing. Transactions are encrypted with a public key and ordered by a committee. Decryption requires a specific number of committee members to collaborate. This method ensures that only when a majority agrees can transactions be decrypted.
‍
For more information read the paper:
How to Share a Secret

How to Securely Replicate Services

The “How to Securely Replicate Services” protocol ensures fair transaction ordering in the context of encrypted transactions by employing a combination of cryptographic techniques and system design principles. The key aspect of the protocol is its ability to maintain a causal order of client requests while preserving the confidentiality of the transactions. This is achieved through mechanisms that ensure that even if the transaction content is encrypted and thus opaque to the servers, the servers can still enforce a consistent and fair ordering of these transactions.
