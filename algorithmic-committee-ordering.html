Algorithmic Committee Ordering

In algorithmic committee ordering schemes, a committee oversees the transaction ordering. Transactions are sent directly to the committee and the committee agrees upon a fair ordering through consensus. Generally, these approaches can handle less than one third of the committee members being byzantine.

Positives

  • Gooput stays unchanged

Negatives

  • Medium Scope as back-running still possible
  • Bad security as attacks still possible
  • Reduced decentralization
  • Increased costs
  • Jostling increased
  • Significant delay

Methods

Hashgraph

The Hashgraph protocol is a distributed consensus algorithm offering fast, fair, and secure transactions without a central authority. It uses a gossip-about-gossip mechanism, where each node communicates information (gossip) about transactions and other nodes' gossips, creating a Hashgraph. This graph records the history of messages in a network, ensuring that the order of transactions is consistent across all nodes. The protocol employs virtual voting to achieve consensus, allowing nodes to deduce the vote of others, thereby achieving Byzantine fault tolerance efficiently. This method provides quicker and fairer consensus compared to traditional blockchain methods. (TBC)

Quick Order-Fair Broadcast Protocol

The Quick Order-Fair Atomic Broadcast protocol operates through consensus rounds in a Byzantine environment. It utilizes a Byzantine FIFO broadcast channel (BCCH) for consistent message delivery. Messages broadcasted are processed and tracked by local vector clocks, which are signed and exchanged among processes. Accumulation of these signed clocks triggers a Byzantine consensus to agree on a message order matrix, represented as a directed graph. This graph, which includes preferences for message sequencing, identifies and collapses strongly connected components (cyclic dependencies), ensuring fair delivery order by the graph's directed edges. Pompe

The Pompe protocol, introduced in the paper "Byzantine Ordered Consensus without Byzantine Oligarchy", works by enhancing State Machine Replication (SMR) in the context of permissioned blockchains. It introduces a new Byzantine Ordered Consensus primitive to ensure that Byzantine nodes cannot control the order of commands. This protocol is designed to provide ordering linearizability and prevent Byzantine oligarchy, where a few nodes dominate the order of operations. Pompe achieves this by a specific structuring of consensus rounds and client interaction, leading to higher throughput and competitive latency compared to traditional BFT SMR protocols. (TBC)

Wendy

The Wendy protocol enhances fairness in blockchain networks by modifying the transaction ordering process. It operates as an additional layer over standard blockchain protocols, selectively applied to certain transactions. Wendy uses fairness models to prevent manipulation of transaction order, focusing on relative and probabilistic fairness. This approach helps mitigate the potential for intentional reordering or delaying of transactions by network participants. By incorporating these fairness models, Wendy ensures a more equitable transaction sequence, balancing the need for fairness with the inherent performance and latency considerations of blockchain systems. (TBC)

Aequitas

The Aequitas protocol ensures order-fairness in blockchain networks through a two-stage process. In the Broadcast Stage, nodes receive transactions and broadcast them using a FIFO method, creating a consistent view of transaction orderings, crucial for fairness. The Finalization Stage involves no further node communication, as each node computes the final transaction ordering locally from received input orderings. This results in a transaction sequence that reflects their original reception order. (TBC)

Themis

The Themis operates in a partially synchronous environment with a committee of nodes, some of which may be adversarial. It stands out for implementing the batch-order-fairness property, ensuring robust fairness in transaction ordering by dictating that transactions received first by a certain fraction of nodes are ordered accordingly. Its design integrates seamlessly into existing leader-based consensus protocols with minimal changes: replicas send transaction order information to the leader, who then constructs a fair block proposal. This is followed by a verification mechanism for replicas to assess the fairness of the proposal and finalize the transaction ordering. (TBC)
