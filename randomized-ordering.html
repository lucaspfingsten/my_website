Randomized Ordering

Randomized Ordering protocols order transactions in two phases. In the first phase, users send their transactions to a committee. The committee then agrees on an order through consensus. In the second phase, once the order is agreed upon, the committee decrypts the transaction with their threshold signatures.

Positives

  • Good Scope
  • Minimal Jostling
  • No increased delay
  • No decreased goodput

Negatives

  • Increased cost
  • Reduced decentralization
  • Security in case of byzantine committee

HoneyBadgerBFT

The HoneyBadgerBFT protocol operates by allowing nodes in a distributed system to reach consensus even under asynchronous conditions, where there are no guarantees on message delivery times. It employs a novel atomic broadcast algorithm to ensure all honest nodes agree on the same set of transactions, without relying on timing assumptions. This is achieved through a combination of threshold cryptography for confidentiality and reliability, and randomized protocols to cope with Byzantine faults. The result is a protocol that remains live and secure even under unpredictable network conditions, making it highly suitable for environments like cryptocurrency networks. (TBC)

Helix

The Helix protocol is a blockchain-based consensus mechanism that ensures fair transaction ordering. It operates in rounds, with a primary node proposing a block of transactions. These blocks are validated by a committee of nodes, elected randomly to prevent bias. Key features include a correlated sampling scheme for transaction selection and encryption, enhancing transaction confidentiality and resistance to censorship. This design aims to significantly limit the possibility of fairness manipulation, offering a robust, fair, and secure method for transaction ordering in distributed networks. (TBC)

How to Share a Secret

The protocol presented in "How to Share a Secret" involves dividing a secret into multiple parts, where any subset of a predefined number of these parts can be used to reconstruct the secret. It employs polynomial interpolation: a secret is embedded in a polynomial of degree one less than the number of pieces needed to reconstruct it. Each piece of the secret is a point on this polynomial. Knowing fewer pieces than required gives no information about the secret, ensuring security even if some parts are compromised. (TBC)

How to Securely Replicate Services

The method presented in the paper “How to Securely Replicate Services” involves a unique protocol for securely replicating services in distributed systems. It ensures the integrity and availability of services even in the presence of server corruptions and client failures. The protocol achieves this by maintaining a causal order of client requests, preventing security breaches caused by intruders manipulating request sequences. Notably, clients interact with the service using a single public key, without needing to identify individual servers. This approach significantly enhances the reliability and security of distributed systems, offering robust defense against both intrusions and benign failures. (TBC)
